# libutf — Fast Unicode Processing in Pure C

A grab-and-go Unicode library built on compressed DFAs and Ragel -G2
state machines.  Processes UTF-8 directly — no conversion to UTF-16
and back.  No malloc.  No dependencies.  670 KB.

Unicode 16.0.  MIT licensed.

```
git clone https://github.com/brazilofmux/utf.git
cd utf && make && make test
```

## What's in the box

| Module | Header | What it does |
|--------|--------|-------------|
| Color Ops | `color_ops.h` | 60+ single-pass string operations that preserve inline PUA color codes |
| CIE97 | `cie97.h` | Perceptual color distance (sRGB → CIELAB) with Kd-tree nearest-neighbor over xterm-256 palette |
| Grapheme | `grapheme.h` | Extended Grapheme Cluster segmentation (UAX #29) — emoji ZWJ, regional indicators, Hangul |
| NFC | `nfc.h` | Canonical normalization (UAX #15) with Hangul algorithmic composition |
| Collation | `collate.h` | DUCET sort (UTS #10) — multi-level comparison and binary sort keys |
| Tables | `utf_tables.h` | Compressed DFA tables for case mapping, character width, GCB, charset approximation |

## Performance vs ICU 74.2

Benchmarks on the same machine, same test data (`bench/`).  Two
scenarios: **UTF-8 input** (libutf native — no conversion) and
**UTF-16 input** (ICU native — no conversion).

### UTF-8 input (the common case)

| Operation | libutf | ICU 74.2 | Ratio |
|-----------|--------|----------|-------|
| Grapheme segmentation | 948 ms | 5,704 ms | **6.0x faster** |
| Case mapping (toupper) | 1,017 ms | 1,216 ms | **1.2x faster** |
| NFC normalization | 97 ms | 122 ms | **1.3x faster** |
| DUCET collation | 165 ms | 197 ms | **1.2x faster** |

### Core-to-core (each library in its native encoding)

| Operation | libutf (UTF-8) | ICU (UTF-16) | Ratio |
|-----------|----------------|--------------|-------|
| NFC normalization | 97 ms | 94 ms | **~parity** |
| DUCET collation | 165 ms | 112 ms | ICU 1.5x faster |

When both libraries work in their native encoding, NFC is at parity.
ICU's collation engine still wins core-to-core thanks to its
specialized fast-Latin comparison path, but that advantage disappears
when the caller's data is UTF-8.

## Design

**Compressed DFA tables.**  Character classification, case mapping,
normalization properties, and collation element lookup are all driven
by deterministic finite automata.  The DFA consumes UTF-8 bytes
directly — one byte in, one state transition, no intermediate code
point decoding.  Tables are generated by the `gen/` pipeline from
Unicode data files and compressed using state merging, column
deduplication, and RUN/COPY phrase encoding.

**Ragel -G2 state machines.**  The 60+ `co_*` color string operations
are generated by [Ragel](https://www.colm.net/open-source/ragel/) using
the `-G2` (goto-driven) backend.  Each operation is a single-pass state
machine that recognizes PUA color code points inline in the UTF-8 stream
and handles them without stripping/re-inserting.

**PUA color encoding.**  Colors are encoded as Unicode Private Use Area
code points inline in UTF-8 strings:

- BMP PUA (3-byte UTF-8): attributes, 256 indexed colors
- Supplementary PUA (4-byte UTF-8, Plane 15): 24-bit RGB

This means colored strings are valid UTF-8 and can be processed by any
Unicode-aware operation without special handling.

**CIE97 perceptual distance.**  When mapping 24-bit RGB to the xterm-256
palette, the library converts to CIELAB color space (sRGB → linear RGB →
XYZ D65 → CIELAB) and searches a pre-built Kd-tree that cycles through
L\*, a\*, b\* axes.  This gives perceptually accurate nearest-color
results instead of the Euclidean-in-RGB approximation most terminals use.

**No malloc.  No global mutable state.**  All DFA tables are `const`.
All functions write to caller-provided buffers.  Thread-safe by
construction.

## Usage

Link against `libutf.a` and `#include` the headers you need:

```c
#include "utf/color_ops.h"   /* co_toupper, co_render_ansi256, ... */
#include "utf/nfc.h"         /* utf_nfc_normalize, utf_nfc_is_nfc */
#include "utf/collate.h"     /* utf_collate_cmp, utf_collate_sortkey */
#include "utf/grapheme.h"    /* utf_grapheme_next, utf_grapheme_count */
#include "utf/cie97.h"       /* co_nearest_xterm256, co_nearest_xterm16 */
```

Compile:
```
gcc -Iutf/include myapp.c -Lutf -lutf -lm
```

See `examples/` for complete programs.

## Building

```
make            # build libutf.a
make test       # run 347 tests
make examples   # build example programs
make ragel      # regenerate color_ops.c from color_ops.rl (requires Ragel)
make clean
```

## Regenerating tables

The `gen/` directory contains the C++ DFA builders, Perl scripts, and
Unicode 16.0 data files.  You only need this when updating to a new
Unicode version.  See [`gen/README.md`](gen/README.md).

## Project structure

```
include/utf/       Public C headers
src/               Implementation (color_ops.rl, cie97.c, nfc.c, collate.c, grapheme.c)
tables/            Pre-generated compressed DFA tables (ready to compile)
tests/             347-test suite
examples/          Example programs (sort, color lookup, NFC check, ANSI rendering)
bench/             Benchmarks vs ICU
gen/               Code generation pipeline (C++ tools + Perl + Unicode data)
```

Extracted from [TinyMUX](https://github.com/brazilofmux/tinymux), where

single-threaded, latency-sensitive environment with no room for ICU's
42 MB footprint.
